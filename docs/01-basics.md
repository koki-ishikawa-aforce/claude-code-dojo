# ミッション01: 基本対話とコードベース理解

## ゴール

- Claude Code に初めてのコードベースを説明させ、読解の相棒として使えるようになる
- 「良い頼み方」の3原則を知る

## 背景解説

Claude Code の最初の実用価値は「コードを書くこと」ではなく「**コードを読むこと**」です。配属直後の案件、他人が書いたモジュール、数年前のレガシー——人間が読むと数時間かかるコードの全体像を、数分で把握する手伝いをしてくれます。

そして、ここが Dojo 全体で一番大事なところですが、**Claude Code の成果はあなたの頼み方でほぼ決まります**。同じタスクでも、頼み方次第で結果は天と地ほど変わります。

### 良い頼み方の3原則

1. **コンテキストを渡す** — 「何のために」「どこを対象に」を言う。Claudeはあなたの頭の中を読めない
2. **タスクを分割する** — 「アプリ全部リファクタして」ではなく、確認可能な単位に区切る
3. **曖昧さを排除する** — 「いい感じに」「ちゃんと」は禁句。期待する結果を具体的に言う

### 良い頼み方 / 悪い頼み方

> ❌ 「このアプリについて教えて」
>
> ✅ 「このアプリの構造を教えて。src配下の各ファイルの役割と、データがどう流れるかを知りたい。これから勤怠集計まわりを触る予定」

悪い例でも何か答えは返ってきます。しかし「何のために知りたいか」を渡した方が、あなたに必要な情報に絞った答えになります。

## ミッション手順

`app/` ディレクトリで `claude` を起動して、以下を順に試してください。

### 1. 構造を説明させる

```
このアプリの構造を教えて。src配下の各ファイルの役割と、
データの流れ（CLI/API → サービス → DB）を知りたい。
```

### 2. 特定の処理を深掘りする

```
月次サマリーはどうやって計算してる？関係する関数を教えて。
休憩時間の控除ルールも知りたい。
```

> この時点で Claude が仕込みバグを列挙して「修正しましょうか？」と提案してくることがあります。**ここでは「直して」と言わないでください**。修正はミッション02で plan mode とセットで体験します。ネタバレされても、自分の目でファイルを確認する価値は変わりません。

### 3. 仕様と実装を突き合わせる

```
SPEC.md の仕様とsrcの実装で、食い違っているところはない？
```

→ Claude は何か見つけるかもしれません。**見つけた内容をそのまま信じず、指摘されたファイルを自分で開いて確認**してください。これが「検証の習慣」の第一歩です（本格的にはミッション02で扱います）。

### 4. 分からない用語を聞く

コードや説明に分からない用語（例: ルーター、prepare文）が出てきたら、遠慮なくその場で聞いてください。

```
さっきの説明の「prepare文」って何？初心者向けに説明して。
```

## チェックポイント

- [ ] アプリの構造説明を受け、自分の言葉で「CLIとAPIとサービス層の関係」を説明できる
- [ ] 休憩控除の計算がどのファイルの何という関数にあるか言える
- [ ] Claudeの説明のうち1つを、実際にファイルを開いて自分の目で確認した

✅ できたらチームチャンネルに「Dojo 01 クリア + アプリ構造を一言で」を投稿！

## 発展課題

- 「このコードベースで改善すべき点を3つ挙げて、重要度順に並べて」と頼み、納得できるか自分の目で確かめる
- `Esc` で Claude の生成を途中で止められることを試す（長い説明が始まったら止めて「もっと簡潔に」と言い直す）

## ハマりどころ

| 症状 | 対処 |
|---|---|
| 説明が長すぎて読めない | 「3行で」「表で」など出力形式も指定してよい |
| 説明が的外れ | 頼み方に目的が入っているか見直す。会話が迷走したら `/clear` で仕切り直し |
| 嘘っぽい説明をされた | ファイル名・行番号を聞いて自分で開く。「その根拠のコードを見せて」も有効 |
