# ミッション02: バグ修正と plan mode — 「検証の習慣」を身につける

## ゴール

- 失敗しているテストを Claude Code に修正させる
- plan mode で「計画 → 承認 → 実装」の流れを体験する
- **Claude の修正を diff とテストで検証してから受け入れる**習慣を身につける

## 背景解説

このミッションが Dojo の心臓部です。理由は、実務で一番危険な使い方が「**AIの修正を読まずに受け入れてマージする**」ことだからです。

Claude Code は優秀ですが、間違えます。もっともらしい修正で別の場所を壊すこともあります。プロの使い方とは、AIを疑うことではなく、**AIの成果物を人間の成果物と同じ基準でレビューする**ことです。あなたが後輩のPRをレビューするのと同じです。

### plan mode とは

`Shift+Tab` で切り替えられる読み取り専用モードです。Claude はコードを調査して**修正計画だけ**を提示し、あなたが承認するまで一切変更しません。「どう直すつもりか先に見たい」ときに使います。修正が1行でも、影響範囲が読めないうちは plan mode から入るのが安全です。

## ミッション手順

### 1. まず自分でテスト結果を見る

```bash
cd app
npm test
```

`buildMonthlySummary > 実働時間を合計する` が失敗しているはずです。**expected 915, received 435** — この数字の意味を30秒だけ自分で考えてみてください（分からなくてOK）。

### 2. plan mode で原因調査させる

`claude` を起動し、`Shift+Tab` で **plan mode** に切り替えてから:

```
npm test で「実働時間を合計する」のテストが失敗してる。
expected 915 に対して received 435。
原因を調査して、修正方針を提示して。まだコードは変更しないで。
```

### 3. 計画をレビューする

Claude が原因と修正方針を提示します。承認する前に確認してください:

- 原因の説明に納得できるか？（ヒント: 435 = 2日目だけの実働時間）
- 修正箇所は1箇所か、複数ファイルに及んでいないか？

納得したら承認して実装させます。

### 4. 修正を検証する（最重要ステップ）

Claude が「修正しました。テストも通りました」と言っても、**そこで終わりにしない**こと。

```bash
git diff        # ← 何が変わったか自分の目で読む
npm test        # ← 自分でも実行する
```

確認する観点:
- [ ] 変更は原因に対応した最小限か？関係ない場所を触っていないか？
- [ ] テストは全件通ったか？（修正で別のテストが壊れることもある）
- [ ] 変更内容を自分の言葉で説明できるか？

### 5. ここまでをコミットする

検証が済んだ修正は、ここでコミットして区切ります:

```bash
git add -A && git commit -m "fix: 月次サマリーの合計計算を修正"
```

後続のミッションで「変更の取り消し」（04）や worktree（08）を使うため、**完了した修正を未コミットのまま持ち越さない**のが安全です。これは実務でも同じです。

### 良い頼み方 / 悪い頼み方

> ❌ 「テスト通るようにして」
>
> ✅ 「このテストが失敗してる原因を調査して修正して。**テスト側を書き換えるのではなく**、実装のバグを直して」

「テスト通るようにして」と頼むと、AIは文字どおり受け取って**テストの期待値の方を書き換える**ことがあります。ゴールの言語化を雑にすると、雑に達成されます。

## チェックポイント

- [ ] plan mode で計画を確認してから実装させた
- [ ] `git diff` を読んで、変更を自分の言葉で説明できる
- [ ] `npm test` を自分で実行し、全件パスを確認した

✅ できたらチームチャンネルに「Dojo 02 クリア + バグの原因を一言で」を投稿！

## 発展課題

- 修正後、「この関数に対して境界値のテストケースを追加して」と頼み、追加されたテストの妥当性をレビューする
- わざと「テスト通るようにして」とだけ頼んだらどうなるか、別ブランチで試してみる（`git checkout -b experiment` してから）

## ハマりどころ

| 症状 | 対処 |
|---|---|
| plan mode のまま実装が始まらない | plan mode は読み取り専用。計画承認後、通常モードに戻って実装させる |
| 修正が大掛かりすぎる | 「最小限の修正にして。リファクタリングは今回は不要」と明示する |
| diff の読み方が分からない | それも Claude に聞いてよい。「この diff を初心者向けに解説して」 |
| テスト通過＝正しいと思ってしまう | ミッション03で「テストが通っていてもバグはある」を体験します |
