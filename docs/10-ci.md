# ミッション10: CI 連携 — PR を自動レビューさせる（発展）

> **前提**: この章は GitHub にリポジトリを push できる人向けの発展章です。会社のコードでは必ずセキュリティポリシーを確認してから行ってください（このDojoリポジトリを個人アカウントに push して試すのが安全です）。

## ゴール

- GitHub Actions 上で Claude Code を動かし、PR の自動レビューを体験する
- 「手元のAI」から「チームのAI」への発想の広がりを知る

## 背景解説

ここまでの Claude Code は「あなたの手元」で動いていました。Claude Code は CI 上でも動かせます。代表的なのが **Claude Code GitHub Actions** で、これを入れると:

- PR に `@claude` とコメントするだけでレビューや修正を依頼できる
- PR 作成時に自動でコードレビューを走らせられる

手元との最大の違いは「**人を選ばない**」ことです。ミッション01〜09で学んだ頼み方のスキルがないメンバーの PR にも、一定品質のレビューが必ず付きます。個人スキルをチーム資産に変える仕組みです。

## ミッション手順

### 1. Dojo リポジトリを自分の GitHub に push する

```bash
cd ~/claude-code-dojo
gh repo create claude-code-dojo --private --source=. --push
```

（gh CLI が未設定なら `gh auth login` から。これも Claude に「ghのセットアップ手伝って」でOK）

### 2. Claude Code GitHub Actions をセットアップする

手元の Claude Code で:

```
/install-github-app
```

対話に従ってアプリのインストールと `ANTHROPIC_API_KEY` のシークレット設定を行います。ワークフローファイル（`.github/workflows/claude.yml`）が作られます。

> `/install-github-app` が使えない環境では、「Claude Code GitHub Actions の公式ドキュメントを調べて、このリポジトリにワークフローを設定して」と頼んでください。

### 3. わざと怪しい PR を作る

```
新しいブランチで、calcBreakMinutes の休憩控除を「常に0を返す」に変更して、
「休憩控除の一時無効化」というタイトルでPRを作って
```

### 4. `@claude` にレビューさせる

GitHub の PR 画面でコメントします:

```
@claude この変更をレビューして。SPEC.md との整合性を必ず確認して
```

Actions が走り、仕様違反（休憩控除ルールの逸脱）が指摘されることを確認してください。

### 5. 片付け

PR をクローズし、ブランチを削除しておきます。

### 良い頼み方 / 悪い頼み方（CI設計にも同じ原則）

> ❌ 自動レビューのプロンプトが「良いコードかレビューして」
>
> ✅ 「SPEC.md との整合性・テストの有無・境界値の考慮、の3観点でレビューして」

CI に仕込むプロンプトは、チーム全員の頼み方を代行するものです。ミッション03で学んだ「観点の指定」がそのまま品質に直結します。

## チェックポイント

- [ ] `@claude` メンションで PR レビューが返ってきた
- [ ] レビュー観点をカスタマイズしたワークフローの意義を説明できる

✅ できたらチームチャンネルに「Dojo 10 クリア（全ミッション制覇🎉） + 一番実務に効きそうな機能」を投稿！

## 発展課題

- PR オープン時に自動レビューが走るワークフローに改造する
- レビュー観点をチームの実際のレビュー指摘傾向に合わせてチューニングする
- hooks（ミッション06） + CI（この章）で「手元で防ぐもの/CIで防ぐもの」の役割分担表を作る

## ハマりどころ

| 症状 | 対処 |
|---|---|
| Actions が起動しない | ワークフローの trigger 設定と、アプリのインストール先リポジトリを確認 |
| APIキーのエラー | リポジトリの Secrets に `ANTHROPIC_API_KEY` が入っているか確認 |
| 会社リポジトリで試していいか不明 | 試さない。まず個人リポジトリで。会社導入はセキュリティ確認が先 |
