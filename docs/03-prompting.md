# ミッション03: 頼み方の改善 — わざと失敗して、指示で挽回する

## ゴール

- 曖昧な指示と具体的な指示で、結果がどれだけ変わるかを体験する
- 「テストが通っていてもバグはある」ことを知る
- 仕様書をコンテキストとして渡すテクニックを覚える

## 背景解説

この題材アプリには、**テストが全部通っているのに仕様と食い違っているバグ**がまだ残っています（ミッション02を終えた人も、です）。

このミッションでは、そのバグをわざと2回探させます。1回目は曖昧な指示で、2回目は具体的な指示で。同じClaude、同じコード、違うのはあなたの指示だけです。

なぜこの練習をするのか。実務でAIの結果がイマイチだったとき、「AIが使えない」で終わらせる人と、「指示のどこが悪かったか」を考えられる人では、半年後の生産性がまるで違うからです。

## ミッション手順

### 1. 【1回目】曖昧に頼んでみる

新しいセッション（`/clear` するか claude を起動し直す）で、あえてこう頼んでください:

```
このアプリにバグがないか見て
```

結果を観察してください。モデルの性能によっては、曖昧な指示でも全部のバグを見つけてくることがあります（最近のモデルほどその傾向があります）。ここでの学びは「見つかるか見逃すか」ではなく、**焦点・確実性・検証観点の差**です。2回目の具体的な指示と見比べると、狙った箇所をピンポイントで、根拠と検証観点つきで返してくる違いが分かります。

### 2. 【2回目】仕様書を渡して具体的に頼む

`/clear` で会話をリセットしてから:

```
SPEC.md の「休憩時間の自動控除ルール」を読んで、
src/utils/time.ts の calcBreakMinutes の実装が仕様どおりか検証して。
特に境界値（ちょうど6時間、ちょうど8時間）に注目して。
テストが通っていても仕様と食い違っていれば指摘して。
```

今度は境界値バグ（`>=` と `>` の食い違い）を確実に見つけてくるはずです。

### 3. 何が違ったかを言語化する

2つの指示を見比べてください。2回目の指示がやったこと:

- **正解の基準を渡した**（SPEC.md という照合先を指定）
- **範囲を絞った**（アプリ全体 → 特定の関数）
- **観点を指定した**（境界値）
- **思い込みを外した**（「テストが通っていても」）

### 4. 修正させて、テストで穴を塞ぐ

```
仕様に合わせて修正して。
あわせて、この境界値バグを二度と見逃さないためのテストケースを追加して。
```

ミッション02で身につけた検証をここでも: `git diff` を読む → `npm test` を自分で実行 → 追加されたテストが「ちょうど360分」を検証しているか確認。

検証できたら、ミッション02と同じくここでコミットして区切ります（後続ミッションの取り消し操作や worktree で、未コミットの修正が巻き添えにならないように）:

```bash
git add -A && git commit -m "fix: 休憩控除の境界値バグを修正"
```

### 良い頼み方 / 悪い頼み方（このミッションの核心）

> ❌ 「バグがないか見て」 — 正解の基準がAI任せになる
>
> ✅ 「SPEC.mdと実装を突き合わせて、特に境界値を検証して」 — 正解の基準・範囲・観点を渡す

AIへの指示は、**優秀だが事情を何も知らない応援メンバーへのタスク依頼**と同じです。事情を知らない人に「バグ見といて」と言うリーダーはいませんよね。

## チェックポイント

- [ ] 曖昧な指示と具体的な指示の結果の違いを体験した
- [ ] 境界値バグ（拘束ちょうど6時間の休憩控除）を修正し、テストを追加した
- [ ] 2回目の指示が良かった理由を、上記4点で説明できる

## 発展課題

- まだ残っている3つ目のバグ（APIまわり、SPEC.mdの「REST API」の節がヒント）を、自分で設計した指示で見つけて修正する。`npm run seed` してから `PORT=3210 npm run dev` で起動し、`curl http://localhost:3210/api/summary/2026-07` を試すと現象が見えます
- 自分の実案件で最近AIに出した指示を1つ思い出し、4つの観点で書き直してみる

## ハマりどころ

| 症状 | 対処 |
|---|---|
| 1回目の曖昧指示で全バグが見つかってしまった | ある意味おめでとうございます。それでも「毎回確実に見つけさせる」には具体的指示が必要、が学びどころ |
| `/clear` せず2回目を頼んでしまった | 1回目の会話が文脈に残っていて比較になりません。`/clear` してやり直し |
| 修正時にテストの期待値の方を変えられた | ミッション02の教訓。「実装を仕様に合わせて。テストは追加のみ」と指示 |
