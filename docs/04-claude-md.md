# ミッション04: CLAUDE.md とコンテキスト管理

## ゴール

- CLAUDE.md でプロジェクトの前提知識・ルールを Claude に仕込み、挙動の変化をビフォーアフターで確認する
- コンテキスト（会話の記憶）の仕組みと、`/clear` `/compact` の使いどころを理解する

## 背景解説

### CLAUDE.md とは

プロジェクトルートに置く Markdown ファイルで、**セッション開始時に毎回自動で読み込まれる**指示書です。ここに書いた内容は、毎回プロンプトで言わなくても常に効いています。

書くべきもの:
- プロジェクトの前提（何のアプリか、どこに何があるか）
- 毎回守ってほしいルール（コーディング規約、応答言語、テストコマンド）
- よく使うコマンド

書くべきでないもの:
- 長大な仕様書の全文（必要なとき「SPEC.mdを読んで」と言えばよい。常時読み込みはコンテキストの無駄遣い）
- 一時的な作業メモ

### コンテキストとは

Claude との会話は「コンテキストウィンドウ」という有限の記憶領域に載っています。会話が長くなると、古い内容が要約されたり、応答の質が下がったりします。

| コマンド | 何をするか | 使いどころ |
|---|---|---|
| `/clear` | 会話を完全リセット | 別のタスクに移るとき。**タスクごとに /clear が基本** |
| `/compact` | 会話を要約して圧縮 | 同じタスクを続けたいが会話が長くなったとき |

「1セッション = 1タスク」を守るだけで、体感品質が大きく上がります。逆に、朝から夕方まで同じセッションで雑多な依頼を続けるのはアンチパターンです。

## ミッション手順

### 1. 【ビフォー】現在の挙動を記録する

`app/` で新しいセッションを開始し:

```
tests/time.test.ts に formatMinutes のテストを1ケース追加して
```

Claude の応答スタイル（言語、説明の量、テストの書き方）を軽くメモしておいてください。確認したら、その変更は捨てます:

```
今の変更を取り消して
```

（`git diff` で本当に戻ったか確認するのを忘れずに）

### 2. CLAUDE.md を書く

`app/CLAUDE.md` を自分で作成します。内容は自分の言葉で書くのが一番勉強になりますが、雛形として:

```markdown
# 勤怠集計アプリ

勤怠の記録と月次集計を行う CLI + REST API。仕様の正は SPEC.md。

## コマンド
- テスト: `npm test` / lint: `npm run lint` / 起動: `PORT=3210 npm run dev`

## ルール
- 応答は日本語。説明は結論から簡潔に
- テストコードのdescribe/itは日本語で書く
- コード変更後は必ず `npm test` と `npm run lint` を実行して結果を報告する
- SPEC.md と食い違う実装を見つけたら、修正前に必ず報告する
```

### 3. 【アフター】同じ依頼を再実行する

**`/clear` してから**（重要:CLAUDE.mdはセッション開始時に読まれる）、ステップ1とまったく同じ依頼をします。

観察ポイント:
- 依頼していないのに `npm test` と `npm run lint` を実行して報告してきたか（**一番確実に差が出るポイント**）
- テストの describe/it が日本語になったか
- 応答が結論から簡潔になったか

> 2つ目・3つ目は差が出ないことがあります。既存テストの describe/it がもともと日本語なので CLAUDE.md なしでも周辺に合わせて日本語で書きますし、グローバルの `~/.claude/CLAUDE.md` で応答言語を設定済みの人も差が見えません。その場合は lint の自動実行で判断してください。

### 4. ルールを1つ育てる

自分が「毎回言うのが面倒」と感じたことを1つ、CLAUDE.md に追記して効果を確かめてください。これが実案件での CLAUDE.md 運用の基本サイクルです。

### 良い頼み方 / 悪い頼み方

> ❌ 毎回のプロンプトで「日本語で。テストも実行して。lintも。」と書き続ける
>
> ✅ 毎回言っていることに気づいたら CLAUDE.md に移す

CLAUDE.md は「**毎回言っていることの置き場**」です。逆に、今回しか関係ない指示はプロンプトに書きます。

## 発展課題

- `~/.claude/CLAUDE.md`（ユーザーグローバル設定）と `app/CLAUDE.md`（プロジェクト設定）の使い分けを Claude 自身に聞いて整理する
- 実案件のリポジトリに置く CLAUDE.md の下書きを作ってみる（コマンド、規約、触ってはいけない場所）

## ハマりどころ

| 症状 | 対処 |
|---|---|
| CLAUDE.md を書いたのに効かない | セッション途中の作成は読み込まれない。`/clear` または再起動 |
| ルールを大量に書いたら一部無視される | ルールは厳選する。多すぎる指示は薄まる。「絶対に守るもの」だけ残す |
| ビフォーアフターの差が分からない | ルールが元の挙動と同じ可能性。わざと分かりやすいルール（例: 応答の冒頭に🎯を付ける）で実験してみる |
