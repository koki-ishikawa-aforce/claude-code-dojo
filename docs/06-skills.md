# ミッション05: 自作 skill を作る

## ゴール

- `.claude/skills/` に自作の skill を作り、`/コマンド` として呼び出す
- description による「Claudeが自動で使う」挙動を体験する

## 背景解説

### skill とは

**繰り返し使う手順書をファイル化したもの**です。「毎回同じ指示をコピペしてる」「CLAUDE.mdのこの節、手順書になってきたな」と思ったら skill の出番です。

CLAUDE.md との違いは読み込まれ方です。CLAUDE.md は毎セッション常に読み込まれますが、skill の本文は**使われるときだけ**読み込まれます。長い手順書をコストゼロで待機させておけます。

### 置き場所と形式

```
.claude/skills/
└── review-jp/
    ├── SKILL.md      ← 本体（frontmatter + 手順）
    └── checklist.md  ← 補助ファイルも置ける（任意）
```

SKILL.md の形式:

```markdown
---
name: review-jp
description: コードレビューを日本語で行うときに使う
---

（ここに手順や指示を書く）
```

呼び出し方は2通り:
1. **あなたが呼ぶ**: `/review-jp` と入力
2. **Claudeが自動で使う**: description を見て「今の作業に関係ある」と判断したとき

### 旧形式 `.claude/commands/` について

以前は `.claude/commands/deploy.md` という単一ファイル形式（カスタムスラッシュコマンド）でした。**現在は skills に統合されています**。旧形式も動き続けますが、これから作るなら skills 一択です（補助ファイル同梱・自動起動・呼び出し制御など機能が上位互換）。社内で古い記事を見たらこの点に注意。

## ミッション手順

### 1. レビュー skill を作る

`app/.claude/skills/review-jp/SKILL.md` を作成:

```markdown
---
name: review-jp
description: 現在の変更（git diff）を日本語でコードレビューする
---

git diff（未コミットの変更）を対象にコードレビューを行う。

## 手順
1. `git diff` で変更内容を取得する
2. 以下の観点でレビューする
   - バグ・ロジックの誤り（最優先）
   - SPEC.md との整合性
   - 命名・可読性
3. 指摘は重要度順に、日本語で報告する
4. 問題がなければ「指摘なし」と明言する

## 出力形式
- 各指摘: `[重要度] ファイル名:行数 — 指摘内容と修正案`
```

### 2. 手動で呼び出す

何か小さい変更を作ってから試します。

```
src/cli.ts の summary の出力に「未退勤: N件」の行を追加して
```

> **注意**: seed データに対して `npm run cli -- summary 2026-07` を実行すると、ミッション03発展課題のバグ（未退勤レコードの非除外）の影響で例外で落ちることがあります。動作確認は `list` コマンドや `npm test` で十分です（落ちたら、それ自体がバグ発見です）。

変更ができたら:

```
/review-jp
```

skill の手順どおりにレビューが走ることを確認してください。

> `/review-jp` は diff 全体の読解・SPEC 照合・テスト実行まで行うため、**数分かかることがあります**。止まっているわけではないので待ってください。

### 3. 自動起動を観察する

`/clear` してから、skill 名を出さずに頼んでみます:

```
さっきの変更をレビューしてほしい
```

description（「コードレビューする」）にマッチして、Claude が自動で skill を使うか観察してください。使われない場合もあります——description の書き方で自動起動の精度が変わることも学びのうちです。

### 4. 自分の業務用 skill を1つ設計する

自分が繰り返しやっている作業を1つ選んで skill 化してください。例:

- `/commit-msg` — 変更内容からコミットメッセージ案を規約どおりに作る
- `/spec-check` — 実装とSPEC.mdの突き合わせ（ミッション03の指示の skill 化!）
- `/explain-pr` — 変更内容を非エンジニア向けに説明する

### 良い頼み方 / 悪い頼み方（skill設計にも同じ原則）

> ❌ description: 「便利なレビュー」 — いつ使うものか Claude が判断できない
>
> ✅ description: 「現在の変更（git diff）を日本語でコードレビューする」 — 対象と動作が明確

description は「**いつこの skill を使うべきか**」を Claude が判断する材料です。プロンプトの3原則（コンテキスト・範囲・具体性）がそのまま効きます。

## チェックポイント

- [ ] `/review-jp` で skill の手順どおりのレビューが実行された
- [ ] 自分の業務用 skill を1つ作って動かした
- [ ] CLAUDE.md と skill の使い分け（常時 vs 必要時）を説明できる

## 発展課題

- skill に補助ファイル（チェックリスト等）を追加し、SKILL.md から参照させる
- チームで共有する価値のある skill を考え、リポジトリにコミットして配布する（`.claude/skills/` はリポジトリごと共有できる）

## ハマりどころ

| 症状 | 対処 |
|---|---|
| `/review-jp` が候補に出ない | セッション途中に作った skill は認識されない。`/clear` または再起動 |
| 自動起動しない | description を具体的に。それでも確実に使わせたいなら `/名前` で明示呼び出し |
| レビューが長い・終わらない | 変更が溜まっていると数分かかる。対象を絞りたければ「この diff だけ」と範囲を指定 |
| skill が長すぎて動作が不安定 | 1 skill = 1 手順に分割。「なんでも屋 skill」は作らない |
