# ミッション05: セッション管理 — コンテキストを制する

## ゴール

- コンテキストの状態が Claude の出力品質を決めることを、実験で確認する
- コンテキストを「捨てる・圧縮する・復元する・巻き戻す」の4操作で自在に整えられる
- 迷走した会話を、上書きで粘らず「戻ってやり直す」習慣を身につける

## 背景解説

### Claude の賢さ = モデル × コンテキストの質

同じモデルに同じ依頼をしても、応答の質はコンテキスト次第で変わります。無関係な話題や失敗のやり取りで汚れたコンテキストの上では、的外れな応答や迷走が起きやすくなります。ミッション04で学んだ「コンテキストは有限」「1セッション = 1タスク」（`/clear` と `/compact` の使い分けは [04の対比表](04-claude-md.md#コンテキストとは) を参照）は、容量の節約であると同時に**品質の話**です。この章はその原則を守るための操作編——コンテキストを良い状態に保つ道具を一通り体験します。

### コンテキストのライフサイクル: 捨てる・圧縮する・復元する・巻き戻す

| 操作 | 手段 | 使いどころ |
|---|---|---|
| 捨てる | `/clear` | 別タスクに移るとき。汚れが手に負えないとき |
| 圧縮する | `/compact`（指示も付けられる: `/compact 修正方針だけ残して`） | 同じタスクを続けたいが会話が長いとき |
| 復元する | `claude -c`（同じディレクトリの直前を再開）/ `claude --resume`・`/resume`（一覧から選択） | 中断からの再開 |
| 巻き戻す | ダブルEsc / `/rewind`（どちらも同じメニュー） | 汚れる前の分岐点に戻ってやり直すとき |

会話はディレクトリ単位で `~/.claude/projects/` 配下に自動保存されています。ターミナルを閉じても PC を再起動しても消えません。**良いコンテキストは消えない資産**——作り直すのではなく、復元して続きから使えます。

### やり直しには階層がある

- **Esc**: 生成を止める。無駄な長文を最後まで生成させない＝コンテキストをゴミで埋めない
- **Rewind（ダブルEsc または `/rewind`）**: プロンプト送信ごとに自動作成されるチェックポイントへ戻る。「会話のみ / コードのみ / 両方」を選べる
- **git**: 最終防衛線。ミッション02のコミット習慣

Rewind のチェックポイントは git とは**別物**のスナップショットです。Claude が編集ツールで行った変更だけを追跡するので、Bash コマンド経由の変更（`rm`・`mv` 等）や自分がエディタでした編集は戻せません。恒久的な区切りは git commit で作ります。

## ミッション手順

### 1. 【中心実験】コンテキストをわざと汚して、質の差を見る

ミッション03の「わざと失敗」方式です。`app/` で新しいセッションを開始し、まず**本題と無関係な話題を3〜4往復**続けてください。例:

```
package.json の devDependencies をひとつずつ説明して
```

```
TypeScript と JavaScript の違いを初心者向けに説明して
```

```
このリポジトリの README を英訳したら何文字くらいになりそう？
```

十分に汚れたところで、本題を頼みます:

```
SPEC.md の休憩控除ルールに対応する実装箇所を特定して、仕様との差分がないか確認して
```

応答を軽くメモしておきます（的外れさ・冗長さ・直前の話題への引きずられ、はないか）。

次に、**入力欄を空にして Esc を2回**押してください。Rewind メニューが開きます。雑談を始める前の時点を選んで会話を巻き戻し、**まったく同じ本題**をもう一度頼みます。

2つの応答を見比べてください。差が出たら、それが「コンテキストの質が出力の質を決める」の実物です。

> 差が小さいこともあります（モデルは多少の汚れなら乗り越えます）。それでも原則は変わりません——**毎回乗り越えてくれる保証はない**からです。ミッション04の「1セッション = 1タスク」は、この品質リスクを構造的に避けるための習慣です。

### 2. Esc で生成を止めて言い直す

```
src 配下の全ファイルを1つずつ詳しく説明して
```

長い応答が始まったら **Esc** で止めて:

```
表で簡潔に。ファイル名と役割の1行説明だけ
```

止めるのは時間の節約のためだけではありません。読まない長文を最後まで生成させると、それがコンテキストに残って以降の応答を圧迫します。

### 3. 良いコンテキストは資産 — claude -c で再開する

手順1の調査の続きが残っている状態で、セッションを **Ctrl+D** で終了します。その後:

```bash
claude -c
```

会話が戻ってきたら、続きを頼みます:

```
さっき特定した実装箇所に対するテスト観点を3つ挙げて
```

「さっき」が通じることを確認してください。昼休みや PC 再起動をまたいでも、積み上げた調査コンテキストを捨てて作り直す必要はありません。

### 4. タスクごとのコンテキストを使い分ける — --resume / /resume

別タスクのセッションをもう1つ作ります。新しいセッション（`claude`）で:

```
このプロジェクトの npm scripts を一覧で説明して
```

応答が来たら終了し、今度は一覧から選んで戻ります:

```bash
claude --resume
```

セッション一覧が開くので、手順3のセッションを選んで復帰してください。対話中なら `/resume` でも同じ一覧が開きます。

観察ポイント: 一覧の見出しは会話の冒頭から作られます。「後で戻る」つもりのセッションは、**1通目にタスク名を入れておく**と探しやすくなります。「1セッション = 1タスク」を守っていれば、この一覧がそのままタスク一覧になります。

### 5. /rewind でコードごと巻き戻す

ミッション04の手順1と同じ依頼を、あえてもう一度使います:

```
tests/time.test.ts に formatMinutes のテストを1ケース追加して
```

変更されたら `git diff` で確認します。ミッション04では「今の変更を取り消して」と自然言語でお願いしましたが、今回は頼まずに:

```
/rewind
```

チェックポイント一覧から依頼前の時点を選び、コードと会話を復元します。`git diff` で変更が消えたことを自分の目で確認してください（検証の習慣）。

自然言語での取り消しは Claude がやる＝確率的、`/rewind` は機構による復元＝決定的。ミッション07（hooks）で本格的に学ぶ「**お願いは方針に、強制は機構に**」の先取りです。

### 良い頼み方 / 悪い頼み方

> ❌ 迷走した会話の上に「やっぱりこうして」「違う、そうじゃなくて」を重ね続ける
>
> ✅ 分岐点まで Rewind で戻り、頼み方を変えてやり直す。手に負えなければ `/clear` で仕切り直し

コンテキストに残った失敗のやり取りは、以降の応答も引きずります。上書きで粘るより、汚れる前に戻るほうが早くて確実です。

## 発展課題

- ターミナルを2枚開き、同じ `app/` で「調査どうし」の2セッションを並行で走らせてみる（コンテキストを混ぜないもう一つの方法が「セッションを分ける」です。ただし**同じファイルを2つのセッションに同時に編集させない**こと——衝突します。本格的な並列開発は worktree を使うミッション10で扱います）
- `/compact` に指示を付けて「何を残すか」を自分で選ぶ（例: `/compact 修正方針と対象ファイルだけ残して`）。圧縮後に「さっきの方針は？」と聞いて、残したいものが生きているか確かめる

## ハマりどころ

| 症状 | 対処 |
|---|---|
| 巻き戻したのに同じ失敗をする | 戻るだけでは頼み方が同じ。分岐点で頼み方の3原則（ミッション03）に沿って言い直す |
| `claude -c` で意図しない会話が開く | `-c` は「同じディレクトリの直前」。狙って戻るなら `--resume` で選ぶ |
| `--resume` の一覧に見つからない | セッションは起動ディレクトリに紐づく。`app/` で起動したものは `app/` で探す |
| `/rewind` してもファイルが戻らない | 復元対象（会話のみ/コードのみ/両方）の選択を確認。Bash 経由の変更や自分の手編集は対象外。最後は git で戻す |
| ダブルEsc でメニューが出ない | 入力欄に文字があると「入力クリア」になる。空にしてから2回押す。生成中は1回目が「中断」に使われる |
